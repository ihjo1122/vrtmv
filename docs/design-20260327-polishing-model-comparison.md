## Problem Statement
대학원 on-device 과목 팀 프로젝트. 터치 기반 AR 카메라 + 온디바이스 VLM 앱에서 두 가지 문제:
1. 객체 검출 바운딩박스가 화면과 안 맞고, 인식률이 저조함
2. "on device" 수업 특성상 양자화 수준별 모델 비교 기능이 필요함 (int4 vs int8 vs float16)

궁극적 비전은 메타 글래스 같은 웨어러블에서 돌아가는 시각 이해 AI. 현재는 폰에서 프로토타입.

## What Makes This Cool
교수님과 동기들 앞에서 "터치하면 AI가 설명해주는 AR 앱"을 시연하는데, 양자화 수준별로 속도와 품질 차이를 눈으로 직접 보여줄 수 있음. on-device AI의 트레이드오프를 앱 하나로 체감시키는 것.

## Constraints
- Galaxy S23 Ultra 타겟 (Snapdragon 8 Gen 2)
- MediaPipe LLM Inference API (.task 형식만 지원)
- 현재 모델: Gemma 3 1B IT int4 (~2GB)
- 비양자화 모델은 크기가 수 GB일 수 있음 (저장공간 + 메모리 주의)
- 시연용이므로 안정성이 최우선

## Premises
1. **바운딩박스 문제가 최우선** — 시연 때 터치했는데 박스가 안 맞거나 검출이 안 되면 치명적. 모델 비교보다 이걸 먼저 고쳐야 함.
2. **모델 비교는 양자화 수준 비교가 주목적** — Gemma 계열을 int4/int8으로 비교. int8 .task 파일을 확보할 수 없는 경우 폴백으로 Gemma 3 1B int4 vs Gemma 2 2B int4 (모델 크기 비교)도 허용.
3. **UI는 버튼 분리 방식** — 메인 화면에서 모델별로 다른 AR 카메라 버튼을 둔다.

## Approaches Considered

### Approach A: 최소 변경 (Minimal)
좌표 버그만 고치고, 메인화면에 버튼 2-3개 추가. 모델 파일은 수동 배치.
- 노력: S | 위험: 낮음 | Completeness: 5/10
- 장점: 가장 빠름, 기존 코드 변경 최소
- 단점: 모델 다운로드 자동화 없음, 비교 UI 없음

### Approach B: 완전한 구현 (Complete) — 선택됨
좌표 버그 수정 + ModelRegistry 시스템 + 모델별 자동 다운로드 + 비교 결과 UI
- 노력: M | 위험: 중간 | Completeness: 9/10
- 장점: 시연 시 완전한 파이프라인, 교수님에게 양자화 이해도 어필 가능
- 단점: 작업량 더 많음

### Approach C: 벤치마크 모드 (Benchmark)
B + 벤치마크 화면 추가. 같은 이미지로 모델별 추론 시간/토큰수/응답품질 자동 측정.
- 노력: L | 위험: 중간 | Completeness: 10/10
- 장점: 논문급 비교 데이터, 시연 임팩트 극대화
- 단점: 가장 많은 작업량

## Recommended Approach

**Approach B: 완전한 구현.** 시연에 필요한 모든 것이 들어있고, 마감 압박에서도 실현 가능한 범위.

### Part 1: 바운딩박스 / 검출 품질 수정

> **접근 방식 변경 (Eng Review 외부 의견 반영):**
> 5개 수정을 동시에 적용하지 않고, 먼저 좌표 로깅을 추가하여 실제 기기에서 진짜 원인을 진단한 후 하나씩 수정한다.

**1-0. [최우선] 좌표 디버그 로깅 추가**
- 터치 시 로그 출력: tapPoint(화면좌표), mappedImagePoint(이미지좌표), 검출된 모든 boundingBox, mapToView 결과
- S23 Ultra에서 실행하여 어느 단계에서 좌표가 어긋나는지 확인
- 진단 결과에 따라 아래 1-1 ~ 1-5 중 필요한 것만 수정
- 파일: `CameraViewModel.kt`, `CoordinateMapper.kt`

**1-1. CoordinateMapper 이중 생성 → ViewModel로 통합**
- 현재: CameraScreen.kt + CameraViewModel.onTapDetect()에서 각각 생성
- 수정: ViewModel에서만 mapper를 생성하고, UI는 이걸 그대로 사용
- 파일: `CameraViewModel.kt`, `CameraScreen.kt`

**1-2. ImageAnalysis ↔ Preview 해상도 일치**
- 핵심: viewSize(Compose layout size)가 Preview의 실제 visible image area가 아닐 수 있음
- 수정: Preview의 실제 렌더링 영역(FILL_CENTER 크롭 포함)을 정확히 계산
- 파일: `CoordinateMapper.kt`, `CameraScreen.kt`

**1-3. 프레임 회전 후 좌표 차원 검증**
- 90도 회전 시 bitmap 차원 swap 후 CoordinateMapper에 정확히 전달되는지 확인
- 파일: `ObjectDetectionManager.kt`

**1-4. ImageProxy → Bitmap row padding (진단 후 필요시만 수정)**
- 현재 방식(oversized bitmap + crop)은 표준 패턴이라 실제로 버그가 아닐 수 있음
- 로깅으로 확인 후 문제 시에만 row-by-row 복사로 변경
- 파일: `ObjectDetectionManager.kt`

**1-5. 검출 정확도 개선**
- EfficientDet-Lite2 threshold 0.3 → 0.25
- ImageAnalysis 해상도 1280x960 시도 (레이턴시 >500ms면 640x480 롤백)
- GazeTargetResolver 150px → 화면 크기 비례
- 파일: `ObjectDetectionManager.kt`, `GazeTargetResolver.kt`

### Part 2: 모델 비교 기능

**2-1. ModelInfo 데이터 클래스 + ModelRegistry**
```kotlin
data class ModelInfo(
    val id: String,           // "gemma-1b-int4", "gemma-1b-int8"
    val displayName: String,  // "Gemma 1B (INT4)"
    val fileName: String,     // "gemma3-1b-it-int4.task"
    val downloadUrl: String,  // HuggingFace URL
    val quantization: String, // "int4", "int8", "float16"
    val expectedSizeMB: Int   // 다운로드 진행률 계산용
)
```
- ModelRegistry: 사용 가능한 모델 목록을 중앙 관리
- 파일: 신규 `domain/model/ModelInfo.kt`, `domain/model/ModelRegistry.kt`

**2-2. GeminiNanoEngine 파라미터화**
- 현재: 모델 파일명 "model.task" 하드코딩
- 수정: `loadModel(modelInfo: ModelInfo)` 메서드 추가. ModelInfo 기반으로 모델 로드
- **내부 저장소 경로를 모델별로 분리**: `files/model.task` (단수) → `files/{modelId}.task` (모델별)
  - 기존 `getModelPath()`의 "이미 있으면 스킵" 로직이 다른 모델을 잘못 로드하는 버그 방지
- **동시성 보호**: `loadModel()` 실행 중 추론 호출을 mutex로 차단. 기존 LlmInference를 `close()` → 새 모델 초기화
- **모델 전환은 CameraScreen 밖에서만 수행** — MainScreen으로 돌아가서 다른 모델 버튼 선택
- **모델 초기화 지연 처리**: CameraScreen 진입 시 모델이 아직 초기화 안 됐으면 로딩 스피너 표시 (5-15초 소요)
- 초기화 실패 시 에러 상태 + MainScreen 복귀
- 파일: `GeminiNanoEngine.kt`

**2-3. ModelDownloadManager 확장**
- `downloadModel(modelInfo: ModelInfo)` 메서드 추가
- 모델별 다운로드 경로: `Download/vrtmv/{fileName}`
- 모델별 존재 여부 체크
- **다운로드 전 저장공간 확인**: `StatFs`로 가용 공간 체크, 부족 시 사용자에게 안내 메시지
- 파일: `ModelDownloadManager.kt`

**2-4. IntroScreen — 변경 없음 (기존 동작 유지)**
- 앱 시작 시 기본 모델(int4)만 자동 다운로드하는 기존 동작 그대로 유지
- 추가 모델 다운로드는 MainScreen에서 on-demand로 처리 (2-5 참조)

**2-5. MainScreen 모델별 버튼**
- 현재 "AR Camera" 버튼 1개 → 모델별 버튼으로 확장
- 예: "AR Camera (INT4)", "AR Camera (INT8)", "AR Camera (FP16)"
- 각 버튼 클릭 시 `modelId: String`을 Navigation argument로 전달. CameraViewModel이 ModelRegistry에서 조회
- 모델이 아직 다운로드 안 된 경우 → MainScreen에서 다운로드 진행 (다이얼로그 + 프로그레스) → 완료 후 CameraScreen 진입
- 파일: `MainScreen.kt`, `AppNavHost.kt`

**2-6. CameraScreen에서 모델 정보 표시**
- 상단에 현재 사용 중인 모델명 + 양자화 수준 표시
- VLM 응답 시 추론 시간(ms) 함께 표시
- 파일: `CameraScreen.kt`, `CameraViewModel.kt`

### 비교용 모델 후보 (MediaPipe .task 형식)
| 모델 | 양자화 | 예상 크기 | 출처 |
|------|--------|----------|------|
| Gemma 3 1B IT | int4 | ~900MB | HuggingFace litert-community (현재 사용 중) |
| Gemma 3 1B IT | int8 | ~1.5GB | HuggingFace 변환 또는 직접 변환 |
| Gemma 2 2B IT | int4 | ~1.5GB | HuggingFace litert-community |

참고: float16 모델은 ~4GB+로 S23 Ultra에서도 메모리 부담. int4 vs int8 비교가 현실적.
직접 양자화는 Google AI Edge의 HF→MediaPipe 변환 파이프라인 사용: https://ai.google.dev/gemma/docs/conversions/hf-to-mediapipe-task

## Open Questions
1. **[블로킹] int8 .task 파일 확보** — HuggingFace litert-community에 있는지 확인 필요. 없으면 HF→MediaPipe 변환 파이프라인으로 직접 변환하거나, 폴백으로 Gemma 2 2B int4와 비교 (모델 크기 비교로 전환)
2. float16 모델은 메모리 부담으로 제외 가능성 높음. int4 vs int8 2개 비교가 기본
3. 시연 마감일 미정 — 마감에 따라 Approach C (벤치마크)로 확장 가능
4. 기존 VLM 모드 토글(Off/On)은 모델별 버튼 체제에서 제거하거나, 각 CameraScreen 내에서 VLM On/Off로 유지. Mock 모드는 디버깅 전용으로 보존

## Success Criteria
1. 10개 이상의 서로 다른 객체를 터치했을 때, 8개 이상에서 바운딩박스가 객체를 올바르게 감쌈 (수동 시연 테스트)
2. 메인 화면에서 모델별 AR 카메라를 선택해서 실행 가능
3. 각 모델의 추론 시간이 AR 오버레이에 표시됨
4. 교수님 앞에서 양자화 수준에 따른 속도/품질 차이를 시연할 수 있음

## What I noticed about how you think
- "나중에 메타 글래스 같은 안경에 돌아가는 앱을 만들거야" — 과제 범위를 넘어서 장기 비전을 갖고 있음. 단순히 점수를 위한 프로젝트가 아님.
- "on device 과목이다보니 양자화 하는 과정도 필요하다고 생각" — 수업의 핵심 취지를 정확히 파악하고, 그걸 앱에 녹이려는 판단. 기술적 시연과 학술적 의미를 동시에 잡으려는 감각.
- "인식률이 너무 저조하고 디스플레이에 박스치는 규격도 아예 맞지 않는 상황" — 문제를 구체적으로 알고 있음. 막연히 "안 돼요"가 아니라 정확히 어디가 어떻게 안 되는지 설명.
